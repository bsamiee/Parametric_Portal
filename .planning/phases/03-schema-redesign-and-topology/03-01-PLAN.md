    Build the 2 root schema groups as colocated definitions. Follow section organization from CLAUDE.md -- schemas go in `// --- [SCHEMA] ---` section, exports at file end in `// --- [EXPORT] ---` section.

    **ROOT 1: Protocol schemas in dispatch.ts (~72 LOC addition)**

    Add a `// --- [SCHEMA] ---` section containing these schema definitions (in this order):

    1. `TelemetryContextSchema` -- S.Struct with fields: attempt (S.Int, greaterThanOrEqualTo(1)), operationTag (S.NonEmptyTrimmedString), spanId (S.String, pattern /^[A-Fa-f0-9]{8,64}$/), traceId (same pattern)

    2. `EnvelopeIdentitySchema` -- S.Struct with fields: appId (S.UUID), issuedAt (S.DateFromString), protocolVersion (inline S.Struct { major: S.Int gte 0, minor: S.Int gte 0 }), requestId (S.UUID), runId (S.UUID), sessionId (S.UUID)

    3. `FailureReasonSchema` -- S.Struct with fields: code (S.NonEmptyTrimmedString), failureClass (S.Literal('retryable', 'correctable', 'compensatable', 'fatal')), message (S.NonEmptyTrimmedString)

    4. `IdempotencySchema` -- S.Struct with fields: idempotencyKey (S.String, pattern /^[A-Za-z0-9:_-]{8,128}$/), payloadHash (S.String, pattern /^[a-f0-9]{64}$/)

    5. `HandshakeEnvelopeSchema` -- S.Union of 3 S.Struct members for init/ack/reject, each with _tag: S.Literal('handshake.init'|'handshake.ack'|'handshake.reject') plus variant-specific fields. Reference the existing kargadan-schemas.ts for exact field definitions.

    6. `CommandEnvelopeSchema` -- S.Struct with _tag: S.Literal('command'), deadlineMs, idempotency (S.optional(IdempotencySchema)), identity (EnvelopeIdentitySchema), objectRefs (S.optional(S.Array(inline S.Struct for SceneObjectRef))), operation (S.Literal with all 13 operation strings), payload (S.Unknown), telemetryContext (TelemetryContextSchema), undoScope (S.optional(S.NonEmptyTrimmedString))

    7. `ResultEnvelopeSchema` -- S.Struct with _tag: S.Literal('result') plus all existing fields from kargadan-schemas.ts

    8. `EventEnvelopeSchema` -- S.Struct with _tag: S.Literal('event') plus all existing fields. eventType field uses S.Literal with all 10 event type strings inlined (NOT a separate const).

    9. `HeartbeatEnvelopeSchema` -- S.Struct with _tag: S.Literal('heartbeat'), identity, mode (S.Literal('ping', 'pong')), serverTime, telemetryContext

    10. `CommandAckSchema` -- S.Struct with _tag: S.Literal('command.ack'), requestId (S.UUID). NOTE: RESEARCH.md row 26 marks this as "DELETE -- only used in C# serialization" but that entry is superseded by this plan. CommandAckSchema is defined here because it is a member of OutboundEnvelopeSchema (item 12) which is decoded by socket.ts for type safety. The research's own Final Schema Placement table (line 708) correctly includes CommandAckSchema in dispatch.ts.

    11. `InboundEnvelopeSchema` -- S.Union(HandshakeEnvelopeSchema, HeartbeatEnvelopeSchema, EventEnvelopeSchema, ResultEnvelopeSchema) -- most specific first

    12. `OutboundEnvelopeSchema` -- S.Union(HandshakeEnvelopeSchema, CommandEnvelopeSchema, HeartbeatEnvelopeSchema, CommandAckSchema)

    Use explicit `S.Struct({ _tag: S.Literal('value'), ... })` NOT `S.TaggedStruct` (per research recommendation -- no make() convenience needed for wire protocol).

    All literal values (FailureClass, CommandOperation, EventType, SceneObjectType, HeartbeatMode) are INLINED into their parent schemas. No standalone `const XSchema = S.Literal(...)` at module level.

    Export only schemas that other files need to import:
    - TelemetryContextSchema (used by checkpoint.ts)
    - EnvelopeIdentitySchema (used by config.ts, agent-loop.ts)
    - FailureReasonSchema (used by agent-loop.ts)
    - CommandEnvelopeSchema (used by config.ts, agent-loop.ts)
    - ResultEnvelopeSchema (used by agent-loop.ts)
    - EventEnvelopeSchema (used by agent-loop.ts)
    - HeartbeatEnvelopeSchema (used by dispatch.ts consumers internally -- check if needed externally)
    - InboundEnvelopeSchema (used by socket.ts)
    - OutboundEnvelopeSchema (used by socket.ts)

    **ROOT 2: Persistence schemas in checkpoint.ts (~32 LOC addition)**

    Add a `// --- [SCHEMA] ---` section containing:

    1. Import `TelemetryContextSchema` from `'../protocol/dispatch'` -- this is the ONE justified cross-file schema import.

    2. `RunEventSchema` -- S.Struct with 11 fields (appId, createdAt, eventId, eventType, idempotency as inline optional S.Struct, payload, requestId, runId, sequence, sessionId, telemetryContext reusing imported TelemetryContextSchema)

    3. `RunSnapshotSchema` -- S.Struct with 6 fields (appId, createdAt, runId, sequence, snapshotHash, state)

    4. `RetrievalArtifactSchema` -- S.Struct with 10 fields (appId, artifactId, artifactType as S.Literal('decision', 'constraint', 'fact', 'verification', 'incident'), body, createdAt, metadata, runId, sourceEventSequence, title, updatedAt)

    These schemas are NOT exported -- they are consumed locally within checkpoint.ts. If agent-loop.ts needs types from these schemas, it accesses them through the CheckpointService interface, not through direct schema import.
