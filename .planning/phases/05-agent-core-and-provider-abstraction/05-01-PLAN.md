           readonly onExecute: (plan: unknown, step: number) => Effect.Effect<{ readonly result: unknown; readonly success: boolean }, unknown, unknown>;
           readonly onVerify:  (result: unknown) => Effect.Effect<{ readonly passed: boolean; readonly details: unknown }, unknown, unknown>;
           readonly onPersist: (checkpoint: unknown) => Effect.Effect<void, unknown, unknown>;
           readonly onDecide:  (verification: unknown, attempt: number, maxCorrections: number) => Effect.Effect<

               unknown, unknown
           >;

    2. **ERRORS:** Define `AgentLoopError`:

    3. **SERVICES:** Define `AgentLoop` as a generic service factory (NOT an Effect.Service -- it's a factory function like `repo()`):
       ```
       const createAgentLoop = (config: AgentConfig) => { ... }
       ```

       The factory returns an object with:
       - `run(intent: string)`: Effect that drives the state machine. Uses `Effect.iterate` starting from `AgentPhase.Plan({ intent })`, with transition logic using `Match.valueTags` on AgentPhase to dispatch to config handlers:
         - `Plan` -> calls `config.onPlan(phase.intent)`, transitions to `Execute`
         - `Execute` -> calls `config.onExecute(phase.plan, phase.step)`, transitions to `Verify`
         - `Verify` -> calls `config.onVerify(phase.result)`, transitions to `Persist`
         - `Persist` -> calls `config.onPersist(phase.checkpoint)`, transitions to `Decide`
         - `Decide` -> calls `config.onDecide(phase.verification, phase.attempt, config.maxCorrections)`, transitions based on decision:
           - `continue` with remaining steps -> `Execute` with step+1
           - `retry` -> `Execute` with same step, attempt+1
           - `correct` -> `Plan` with nextIntent, correctionCycle+1
           - `fail` -> `Failed`
         - `Complete` / `Failed` -> terminates iteration (while predicate returns false)
       - `resume(state: unknown)`: Effect that hydrates from persisted state and re-enters the loop

       **CRITICAL:** The loop is generic. It never imports Kargadan types. The `onPlan`, `onExecute`, etc. handlers are the app's injection points. The `unknown` types in AgentConfig are intentional -- the app provides typed implementations; the loop only orchestrates transitions.

       **Loop termination:** `Effect.iterate` uses a `while` predicate that checks if the phase is `Complete` or `Failed`. All other phases continue iteration.

       **Error propagation:** Each phase handler call is wrapped in `Effect.mapError` to produce `AgentLoopError` with the phase name.

    4. **EXPORT:** Named exports of `AgentPhase`, `AgentLoopError`, `createAgentLoop`, and the `AgentConfig` type.

    **Create `apps/kargadan/plugin/src/CommandCatalogExporter.cs`:**

    **IMPORTANT:** Load the `coding-csharp` skill.

    This class enumerates all registered Rhino commands via the Rhino SDK on plugin load and serializes them as the `catalog` protocol message JSON. Key implementation:

    1. Use `Rhino.Commands.Command.GetCommandNames()` to get all registered command names
    2. For each command, use `Rhino.Commands.Command.LookupCommandName(name)` to get the `Command` object
    3. Extract: name, English name, command style (scripted vs interactive), whether it modifies geometry (isDestructive heuristic: commands with `TransformCommand` or `AddCommand` base types, or commands matching known destructive patterns)
    4. Serialize to JSON matching the `catalog` envelope schema
    5. The exporter is called once after WebSocket handshake completes, sending the full catalog as a single message

    The `isDestructive` heuristic: Mark commands as destructive if they are NOT in the `Rhino.Commands.Style.ScriptRunner` category AND they match known mutation patterns (Move, Copy, Delete, Trim, Split, Fillet, Chamfer, Boolean*, Offset, Scale, Rotate, Mirror, Array*, Extrude, Loft, Sweep, etc.), or if their command style is not `Hidden`. For commands that cannot be classified, default to `isDestructive: true` (safe default -- undo wrapping is cheap).

    The params array will be minimal for this phase -- most Rhino commands don't expose typed parameters via the SDK. Use an empty array for now. Phase 6+ can enhance parameter extraction.
