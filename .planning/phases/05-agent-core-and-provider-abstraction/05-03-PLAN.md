    1. **CONSTANTS:** Define the core system prompt template parts:
       ```
       const _IDENTITY = `You are Kargadan, an AI agent that controls Rhino 9 through natural language. You can create, modify, query, and delete 3D geometry by executing Rhino commands and RhinoCommon API calls through the bridge.` as const;

       const _BEHAVIORAL_RULES = `
       OPERATION PROTOCOL:
       1. PLAN: Analyze the user's request. Search the command knowledge base for relevant commands. Propose a sequence of tool calls.
       2. CONFIRM: Present your plan to the user. Wait for approval before executing any write operations. Read operations execute freely.
       3. EXECUTE: Execute one tool call at a time. Each tool call is atomic -- one command, one undo record.
       4. VERIFY: Check the result. Did the bridge return status 'ok'? Does the result match the intent?
       5. ON FAILURE: Do NOT retry the same approach. Undo the failed attempt. Read the document state. Understand what went wrong. Correct your approach. Maximum 2 correction attempts.

       TOOL USAGE:
       - Read tools (scene summary, object metadata, layer state) are free to call without user confirmation.
       - Write tools (create, update, delete objects) require user approval before execution.
       - Each write tool call carries an undo scope -- the user can Cmd+Z to undo your last action.
       - Never batch multiple commands into a single tool call. One command per step.
       ` as const;

       const _ERROR_HANDLING = `
       WHEN SOMETHING GOES WRONG:
       - A failed tool call means your approach was wrong, not that a retry will fix it.
       - Before your next attempt: (1) undo the failed action, (2) read scene state, (3) understand the failure, (4) formulate a different approach.
       - After 2 correction attempts, explain to the user what you tried and what went wrong. Do not keep retrying.
       ` as const;
       ```

    2. **FUNCTIONS:** Define `buildSystemPrompt`:
       ```
       const buildSystemPrompt = (context: {
           readonly availableTools: ReadonlyArray<{ readonly name: string; readonly description: string; readonly isDestructive: boolean }>;
           readonly sceneContext?: string;
       }) => ...
       ```
       - Combines _IDENTITY, _BEHAVIORAL_RULES, _ERROR_HANDLING
       - Appends available tool list with names, descriptions, and destructive/readonly annotations
       - Appends scene context if provided (Phase 6 will supply the full scene summary; Phase 5 uses a minimal placeholder)
       - Returns a single string

    **Create `apps/kargadan/harness/src/agent/tool-handlers.ts`:**

    Section order: TYPES -> FUNCTIONS -> EXPORT.

    This file provides the Toolkit handler layer that maps tool call invocations to bridge dispatch calls.

    1. **TYPES:** Define the handler builder config:
       ```
       type ToolHandlerConfig = {
           readonly dispatch: CommandDispatch;
           readonly undoScope: string;
       };
       ```

    2. **FUNCTIONS:** Define `buildToolHandlers`:
       ```
       const buildToolHandlers = (
           config: ToolHandlerConfig,
           tools: ReadonlyArray<Tool.Any>,
       ) => ...
       ```
       Steps:
       - Build a handler record where each key is a tool name
       - For each tool, create a handler function that:
         a. Reads the Tool.Destructive annotation to determine if this is a write tool
         b. Builds an `Envelope.Command` from the tool call parameters:
            - `operation`: derive from tool name (map tool name to Operation literal, e.g., "Box" -> "script.run" with payload `{ script: "_Box" }`)
            - `payload`: the tool call arguments
            - `undoScope`: config.undoScope for destructive tools, undefined for read tools
            - `deadlineMs`: from HarnessConfig
            - `requestId`: crypto.randomUUID()
            - `idempotency`: generate for write tools
         c. Calls `config.dispatch.execute(command)` and returns the result
       - Return the handler record suitable for `toolkit.toLayer(handlers)`

       The operation mapping from tool name to Envelope.Command operation. Define an explicit static mapping constant:
       ```
       const _DIRECT_OPERATION_MAP = {
           'read.scene.summary':     'read.scene.summary',
           'read.object.metadata':   'read.object.metadata',
           'read.object.geometry':   'read.object.geometry',
           'read.layer.state':       'read.layer.state',
           'read.view.state':        'read.view.state',
           'read.tolerance.units':   'read.tolerance.units',
           'write.object.create':    'write.object.create',
           'write.object.update':    'write.object.update',
           'write.object.delete':    'write.object.delete',
           'write.layer.update':     'write.layer.update',
           'write.viewport.update':  'write.viewport.update',
           'write.annotation.update':'write.annotation.update',
       } as const satisfies Record<string, typeof Operation.Type>;
       ```
       Resolution logic:
       - If tool name is a key in `_DIRECT_OPERATION_MAP`, use that literal as the operation with the tool call args as payload
       - Otherwise, treat as a Rhino command and dispatch as `'script.run'` operation with `{ script: '_' + toolName }` payload (RunScript invocation convention: underscore prefix suppresses Rhino's command-line echo)
       - Import `Operation` from `../protocol/schemas.ts` to ensure type safety against the Operation literal union

    3. **EXPORT:** Named exports of `buildToolHandlers` and `ToolHandlerConfig`.

    **NOTE on Toolkit.toLayer usage:** The dynamic nature of RAG-selected tools means the toolkit changes each turn. `buildToolHandlers` is called each turn with the current turn's tools to produce a handler record. This is then used with `toolkit.toLayer(Effect.succeed(handlers))` or directly via `toolkit.of(handlers)` depending on the exact @effect/ai API. Verify from installed types.
