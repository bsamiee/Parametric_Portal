    **Create `apps/kargadan/harness/src/agent/loop.ts`:**

    This file creates the Kargadan-specific agent configuration and wires it into `createAgentLoop` from packages/ai. Section order: CONSTANTS -> FUNCTIONS -> SERVICES -> EXPORT.

    1. **CONSTANTS:**
       - `_MAX_CORRECTIONS = 2` (locked decision: original attempt + 2 corrections = 3 total tries). **NOTE:** The existing `HarnessConfig.correctionCycles` in config.ts defaults to 1 (`Config.withDefault(1)`). Update the default to 2 in the config.ts changes below to match this locked decision.
       - `_RAG_TOP_K = 10` (starting value per research recommendation; tunable via config)

    2. **FUNCTIONS:**

       Define each phase handler as a standalone function. These are the Kargadan-specific implementations injected into the universal loop:

       **`_onPlan`:** Takes user intent string. Steps:
       - Call `ragToolSelector` with intent, injecting AiService.searchQuery and toToolManifest adapter
       - Use AiRuntime.generateText with the assembled toolkit + system prompt instructing the LLM to plan the operation
       - System prompt includes: Rhino context, plan-then-confirm instruction, available tool descriptions
       - Return `{ plan: parsedPlan, steps: numberOfSteps }`
       - The plan includes the proposed sequence of tool calls that the agent will present to the user for confirmation before execution

       **`_onExecute`:** Takes plan + step number. Steps:
       - Extract the tool call for this step from the plan
       - Check if the tool is destructive via Tool.Destructive annotation
       - For destructive tools: dispatch via CommandDispatch.execute with undo wrapping (the bridge handles BeginUndoRecord/EndUndoRecord per command per locked decision)
       - For read tools: dispatch freely without undo overhead
       - One tool call per step (locked decision: atomic, debuggable, clean undo)
       - Return `{ result: bridgeResponse, success: status === 'ok' }`

       **`_onVerify`:** Takes execution result. Steps:
       - For Phase 5: basic verification only (did the bridge return status 'ok'?)
       - Phase 7 will add deterministic geometry checks and visual verification
       - Return `{ passed: result.success, details: result }`

       **`_onPersist`:** Takes checkpoint data. Steps:
       - Call Chat.exportJson on the current Chat.Service to get serialized conversation JSON
       - Call AgentPersistenceService.persistCall with the tool call record, checkpoint snapshot, AND chatJson
       - **NOTE:** `AgentPersistenceService.persistCall` (packages/database/src/agent-persistence.ts:115-126) currently hardcodes `chatJson: ''` in the payload. The executor MUST extend `persistCall` to accept a `chatJson: string` parameter and pass it through to the payload instead of the empty-string literal. This is a prerequisite for Chat persistence to work.
       - This replaces the empty-string placeholder with real serialized conversation data

       **`_onDecide`:** Takes verification result + attempt number + maxCorrections. Steps:
       - Use `Match.value` on the verification outcome:
         - Passed -> `{ action: 'continue' }` (proceed to next step or complete)
         - Failed, attempt < maxCorrections -> investigate failure via read tools (query scene state, check what went wrong), then `{ action: 'correct', nextIntent: correctedApproach }`
         - Failed, attempt >= maxCorrections -> `{ action: 'fail' }` (surface to user with what was tried)
       - Before correction: undo the failed attempt (locked decision: "always undo the failed attempt before trying the corrected approach")
       - The correction does NOT blindly retry -- it reads API docs, examines document state, understands what went wrong (locked decision)

    3. **SERVICES:** Define `KargadanAgent` as an Effect.Service that:
       - Yields AiRuntime, AiService, CommandDispatch, AgentPersistenceService
       - Constructs `config: AgentConfig` from the phase handlers above
       - Calls `createAgentLoop(config)` to get the loop instance
       - Exposes `{ run, resume }` from the loop
       - Stores a `Ref<Chat.Service>` for the active chat -- updated after each generateText call

    4. **EXPORT:** Named export of `KargadanAgent`.

    **Extend `apps/kargadan/harness/src/protocol/dispatch.ts`:**

    Add a `receiveCatalog` method to `CommandDispatch`:
    - After successful handshake, call `socket.takeEvent` or a dedicated `socket.takeCatalog` to receive the catalog message
    - The catalog arrives as a single `{ _tag: 'catalog', entries: [...] }` envelope from the bridge
    - Parse via `Envelope.Catalog` schema
    - Return the decoded catalog to the caller (harness.ts)

    Also ensure `execute` supports forwarding tool call commands to the bridge with proper undo scope.

    **Extend `apps/kargadan/harness/src/socket.ts`:**

    Update `_dispatchChunk` to handle the `catalog` tag:
    - Add `{ _tag: 'catalog' }` to the match expression
    - Route catalog messages to a dedicated `Deferred<Envelope.Catalog>` or to the pending map (depending on whether it uses requestId correlation)
    - If the catalog is unsolicited (sent by bridge after handshake without a request), use a dedicated `Deferred` that `receiveCatalog` awaits

    **Extend `apps/kargadan/harness/src/config.ts`:**

    Add new config entries:
    - `ragTopK`: `Config.integer('KARGADAN_RAG_TOP_K').pipe(Config.withDefault(10))`
    - `aiProvider`: `Config.string('KARGADAN_AI_PROVIDER').pipe(Config.withDefault('anthropic'))` -- selects LLM inference provider
    - `aiModel`: `Config.string('KARGADAN_AI_MODEL').pipe(Config.withDefault('claude-sonnet-4-20250514'))` -- selects model

       - Call `dispatch.receiveCatalog()` to get the bridge's command catalog

    2. **Seed knowledge base from catalog:**
       - Convert catalog entries to AiService.seedKnowledge format using the existing manifest adapter
       - Call `ai.seedKnowledge({ entityType: 'command', manifest: catalog.entries, namespace: 'kargadan', scopeId: Option.getOrNull(manifestScopeId) })`
       - This replaces the static `KARGADAN_COMMAND_MANIFEST_JSON` env var approach -- the catalog now comes live from the bridge
       - Keep the env-var seeding as a fallback: if `manifestJson.trim().length > 0`, seed from env; otherwise expect catalog from bridge

    3. **Restore Chat on resume:**
       - When hydrating a session, read the `chatJson` field from the checkpoint
       - Call `AiRuntime.chatFromJson(chatJson)` (added in Plan 01) to restore the Chat.Service with full conversation history
       - If chatJson is empty string or parse fails, start with `AiRuntime.chat()` (fresh Chat.empty)
       - Pass the restored Chat.Service to the KargadanAgent

    4. **Run new agent loop:**
       - Replace the old `AgentLoop.handle(...)` call with `KargadanAgent.run(userIntent)` or `KargadanAgent.resume(sessionId)`
       - The KargadanAgent internally uses createAgentLoop with the Kargadan-specific phase handlers
       - For Phase 5, the user intent comes from a simple stdin prompt (Phase 8 builds the full CLI)

    5. **Persist final Chat state:**
       - After the loop completes, call `Chat.exportJson` on the final Chat.Service
       - Store as chatJson in the final checkpoint via AgentPersistenceService.persistCall

    6. **Layer composition update -- PROV-02 provider wiring:**
       - Add `KargadanAgent.Default` to the layer composition
       - AiService.KnowledgeDefault already provides AiRuntime + AiRuntimeProvider
       - **Wire HarnessConfig provider/model into AiRegistry:**

         c. Call `AiRegistry.layers(settings)` to obtain `{ language, fallbackLanguage, embedding }` Layer records
         d. Compose the resolved `language` Layer (and fallback layers) into the harness Layer composition via `Layer.provide` so that `AiRuntime` receives the correct `LanguageModel` implementation at runtime
         e. The embedding Layer remains OpenAI (fixed per CONTEXT.md decision) -- no override needed
       - This ensures PROV-02: provider selection stored in config is resolved at runtime via AiRegistry, not just dead env-var stubs
