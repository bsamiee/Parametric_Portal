  <action>
**csproj changes:**
1. Override `<TargetFramework>net9.0</TargetFramework>` in a local PropertyGroup (overrides Directory.Build.props net10.0). This is required because Rhino 9 WIP ships .NET 9 runtime — loading a net10.0 assembly causes TypeLoadException.
2. Remove the `<PackageReference Include="Rhino.Inside" .../>` line. Rhino.Inside is for embedding Rhino in external processes (Windows-only); the plugin runs inside Rhino natively and does not need it. Requirement TRAN-02 (user override to Rhino 9 only) confirms single target.

**PortFile.cs:**
Create `PortFile` static class in `ParametricPortal.Kargadan.Plugin.src.transport` namespace.
- `WritePortFile(int port)`: Writes JSON `{"port": N, "pid": N, "startedAt": "ISO8601"}` to `~/.kargadan/port`. Use atomic write pattern: write to `port.tmp`, then `File.Move(temp, target, overwrite: true)` to prevent partial reads. Create `~/.kargadan/` directory if it does not exist.
- `DeletePortFile()`: Best-effort `File.Delete` wrapped in try/catch IOException (cleanup may fail if Rhino force-quits).
- `ReadPortFile()`: Returns `Fin<(int Port, int Pid, DateTimeOffset StartedAt)>` by deserializing the JSON. Used for testing/validation.
- Port file path: `Path.Combine(Environment.GetFolderPath(Environment.SpecialFolder.UserProfile), ".kargadan", "port")`.
- Use `System.Text.Json.JsonSerializer` for serialization. Use `DateTimeOffset.UtcNow` for timestamp.

**ThreadMarshaler.cs:**
Create `ThreadMarshaler` static class in the transport namespace.
- `RunOnUiThread<T>(Func<Fin<T>> operation)`: Returns `Task<Fin<T>>`. Creates a `TaskCompletionSource<Fin<T>>`, calls `RhinoApp.InvokeOnUiThread(new Action(() => { tcs.SetResult(operation()); }))`, returns `tcs.Task`. This ensures all RhinoDoc operations execute on the main thread (macOS AppKit enforces this; violation causes NSInternalInconsistencyException).
- The method accepts a `Func<Fin<T>>` so the caller can pass LanguageExt Fin-returning logic and the result is properly propagated across threads.

**WebSocketHost.cs:**
Create `WebSocketHost` class in the transport namespace.
- Constructor accepts `Action<WebSocket, CancellationToken> onClientConnected` callback (or similar delegate) for message dispatch.
- `Start()`: Binds `TcpListener` to `IPAddress.Loopback` with port 0 (OS assigns available port). Reads assigned port via `((IPEndPoint)listener.LocalEndpoint).Port`. Calls `PortFile.WritePortFile(assignedPort)`. Starts accept loop on a background thread via `Task.Run`.
- Accept loop: `while (!ct.IsCancellationRequested)` — accepts TcpClient, reads HTTP upgrade request from NetworkStream, validates it contains `Upgrade: websocket`, extracts `Sec-WebSocket-Key`, computes RFC 6455 accept hash (`SHA1(key + "258EAFA5-E914-47DA-95CA-C5AB0DC85B11")` base64), sends 101 Switching Protocols response, then creates `WebSocket.CreateFromStream(stream, new WebSocketCreationOptions { IsServer = true, KeepAliveInterval = TimeSpan.FromSeconds(5), KeepAliveTimeout = TimeSpan.FromSeconds(15) })`.
- After WebSocket creation, starts a continuous receive loop (`ReceiveAsync` in while loop) that: (a) accumulates message fragments into a complete message, (b) deserializes JSON to determine message type via `_tag` field, (c) for command messages, calls `ThreadMarshaler.RunOnUiThread` to marshal execution to UI thread, (d) sends response back on WebSocket. The receive loop must remain active at all times — .NET 9 PING/PONG keep-alive only processes incoming PONG frames while ReceiveAsync is outstanding.
- `Stop()`: Cancels the CancellationTokenSource, calls `PortFile.DeletePortFile()`, disposes TcpListener.
- Support single client at a time (one harness connects to one Rhino instance). If a new client connects while one is active, close the old connection.
- Use `LanguageExt.Fin<T>` for error handling in the dispatch path. Log errors via `RhinoApp.WriteLine` (Rhino's command-line output) since the plugin is headless.
- Keep-alive: `.NET 9 WebSocketCreationOptions` handles transport-level PING/PONG. Application-level heartbeat is already handled by the existing `KargadanPlugin.HandleHeartbeat` method — WebSocketHost routes heartbeat messages to it.

**Important constraints per CONTEXT.md:**
- JSON end-to-end as single wire format. Use `System.Text.Json` for all serialization.
- Tagged union with `_tag` discriminator — use `[JsonPolymorphic(TypeDiscriminatorPropertyName = "_tag")]` if creating new wire types, or route through existing `CommandRouter.Decode` which already handles `_tag` dispatch.
- Plugin is fully headless — no status bar, no tray icon, no visible UI.
- Auto-start via `PluginLoadTime.AtStartup` (already set in KargadanPlugin).

Read `csharp-standards` skill rules if available. Follow existing C# patterns in the codebase: LanguageExt `Fin<T>` for error handling, `Option<T>` for optional state, NodaTime `Instant` for timestamps, `[BoundaryAdapter]` attribute on boundary classes, section separators matching CLAUDE.md file organization.

  <action>
Modify `KargadanPlugin` to integrate the WebSocketHost into the plugin lifecycle.

**OnLoad changes:**
1. After creating `BoundaryState` (EventPublisher + SessionHost), create a `WebSocketHost` instance. Pass a message dispatch callback that routes messages through the existing `HandleHandshake`, `HandleCommand`, and `HandleHeartbeat` methods based on the `_tag` field.
2. Call `webSocketHost.Start()` to begin accepting connections.
3. Add `WebSocketHost` to `BoundaryState` record (extend it with a `WebSocketHost` field).
4. Ensure `PluginLoadTime` property returns `AtStartup` — confirm this is already set or override it.

**OnShutdown changes:**
1. Before closing the SessionHost, call `webSocketHost.Stop()` which cancels the accept loop, closes active WebSocket, and deletes the port file.

**Message dispatch callback:**
The WebSocketHost needs to call back into KargadanPlugin for message handling. The callback receives raw deserialized JSON and the active WebSocket for sending responses. The dispatch logic should:
1. Read `_tag` from the JSON.
2. Route `handshake.init` to `HandleHandshake` → serialize response → send on WebSocket.
3. Route `command` to `HandleCommand` (which already decodes via `CommandRouter.Decode`) → serialize response → send on WebSocket.
4. Route `heartbeat` to `HandleHeartbeat` → serialize response → send on WebSocket.
5. For push messages (events from plugin to harness), the existing `EventPublisher.Drain()` pattern already works — WebSocketHost can periodically drain and push, or events can be pushed immediately. Per CONTEXT.md, events are push messages without requestId.

**PluginLoadTime override:**
Ensure the property `public override PluginLoadTime LoadTime => PluginLoadTime.AtStartup;` is present. If not, add it. This ensures the WebSocket listener opens immediately when Rhino starts — per user decision.

**Keep the existing API surface intact** — `HandleHandshake`, `HandleCommand`, `HandleHeartbeat`, `DrainPublishedEvents` still work. The WebSocketHost is an additional transport layer that calls into these methods.
