**port-discovery.ts:**
Create a new module at `apps/kargadan/harness/src/transport/port-discovery.ts`.

Define `PortFileSchema` using `Schema.Struct` with fields: `port` (Schema.Number), `pid` (Schema.Number), `startedAt` (Schema.String — ISO 8601).

Define `readPortFile` as an `Effect.fn('kargadan.portDiscovery.read')` that:
1. Reads `~/.kargadan/port` via `@effect/platform` FileSystem (or `node:fs/promises` wrapped in `Effect.tryPromise`).
2. Decodes JSON via `Schema.decodeUnknown(Schema.parseJson(PortFileSchema))`.
3. Validates PID is alive: check via `process.kill(pid, 0)` wrapped in `Effect.try` — if it throws, the PID is dead. A dead PID means the port file is stale; return a tagged error `PortFileStale`.
4. Returns `{ port, pid, startedAt }` on success.

Define errors:
- `PortFileNotFound` (Data.TaggedError) — file does not exist.
- `PortFileStale` (Data.TaggedError) — file exists but PID is dead.
- `PortFileInvalid` (Data.TaggedError) — file exists but JSON decode failed.

Port file path: `path.join(os.homedir(), '.kargadan', 'port')`.

**reconnect.ts:**
Create a new module at `apps/kargadan/harness/src/transport/reconnect.ts`.

Define the reconnection schedule per CONTEXT.md decisions:
```
const _reconnectSchedule = Schedule.exponential(Duration.millis(500), 2).pipe(
    Schedule.jittered,
    Schedule.intersect(Schedule.recurs(50)),
    Schedule.upTo(Duration.seconds(30)),
);
```
This starts at 500ms, doubles with jitter, caps at 30s, max 50 attempts.

Define `ReconnectionSupervisor` as an `Effect.Service`:
- Maintains `Ref<'disconnected' | 'connected' | 'reconnecting'>` for connection state.
- `supervise(connectOnce: Effect)`: Wraps a connection-establishing effect in retry logic. On disconnect:
  1. Sets state to `'reconnecting'`.
  2. Logs disconnection.
  3. Re-reads port file via `readPortFile` on each retry attempt (plugin may restart on different port).
  4. On successful connection, sets state to `'connected'`.
  5. On `PortFileStale` or `PortFileNotFound`, continues retrying (plugin not yet restarted).
- Per CONTEXT.md: "Drop and notify during disconnection — reject outbound sends with immediate error." The supervisor should expose a method or Ref that the socket client can check to reject sends during disconnection.

**config.ts changes:**
1. Remove `_pluginHost` and `_pluginPort` config entries — port is now discovered dynamically via port file, not via environment variables.
2. Remove `resolveSocketUrl` — replaced by `readPortFile` in port-discovery.ts.
3. Add new config entries:
   - `KARGADAN_RECONNECT_MAX_ATTEMPTS` (default 50)
   - `KARGADAN_CHECKPOINT_DATABASE_URL` (Config.string, required — PostgreSQL connection string)
4. Keep all other config entries unchanged.

**socket.ts changes:**
1. Replace the `KargadanSocketClientLive` layer construction: instead of `HarnessConfig.resolveSocketUrl`, use `readPortFile` from port-discovery.ts to get the port, then construct `ws://127.0.0.1:${port}` dynamically.
2. The socket client should detect disconnection (WebSocket close/error) and surface it to the reconnection supervisor. Use `Effect.catchTag` on `SocketCloseError` to signal disconnection.
3. Per CONTEXT.md: "In-flight commands on disconnect: fail and report with disconnection error. No auto-retry." When the connection is in `'disconnected'` or `'reconnecting'` state, the `request` method should fail immediately with a `DisconnectedError` rather than queuing.

Follow CLAUDE.md conventions: named exports, section separators, `Effect.fn` for traced functions, `Data.TaggedError` for errors, no barrel files, 4-space indentation, `const` only.

**schema.ts:**

Define the checkpoint table schema using `@effect/sql` Model.Class or plain Schema:
- Table name: `kargadan_checkpoints`
- Columns:
  - `session_id` (text, primary key) — identifies the agent session
  - `conversation_history` (jsonb) — serialized array of conversation messages
  - `loop_state` (jsonb) — serialized loop state (current stage, attempt count, pending operations)
  - `state_hash` (text) — SHA-256 hash of canonical state (reuse `hashCanonicalState` from persistence-trace.ts)
  - `scene_summary` (jsonb, nullable) — last known scene state for reconnect verification
  - `sequence` (integer) — monotonically increasing checkpoint number
  - `created_at` (timestamptz) — checkpoint creation time
  - `updated_at` (timestamptz) — last update time

Define schemas for the JSONB columns using Effect Schema:
- `ConversationHistorySchema` — array of message objects (keep flexible with `Schema.Unknown` for now; Phase 5 will define the full Chat schema)
- `LoopStateSchema` — `{ stage: string, attemptCount: number, pendingOperations: number }` (enough to resume the loop)

Export the table name, schemas, and a `CheckpointRow` type derived from the schema.

**checkpoint.ts:**
Create `apps/kargadan/harness/src/persistence/checkpoint.ts`.

Define `CheckpointService` as `Effect.Service` with scoped constructor:
1. Yield `SqlClient.SqlClient` from `@effect/sql`.
2. Define methods:

- `save(checkpoint)`: Upsert into `kargadan_checkpoints` using `sql\`INSERT INTO ... ON CONFLICT (session_id) DO UPDATE SET ...\``. Include `state_hash` computed via `hashCanonicalState` (import from persistence-trace.ts or inline — it is a pure function).

- `restore(sessionId)`: `sql\`SELECT * FROM kargadan_checkpoints WHERE session_id = ${sessionId}\`` → decode via Schema → return `Option<Checkpoint>`. Returns `None` if no checkpoint exists for this session.

- `verifySceneState(sessionId, currentSceneHash)`: Compares `currentSceneHash` against the checkpoint's `state_hash`. Returns `{ diverged: boolean, checkpoint: Option<Checkpoint> }`. Per CONTEXT.md: "Verify on reconnect — harness queries current scene state and compares to checkpoint. Flags divergence if user made manual changes during disconnect."

- `delete(sessionId)`: Deletes the checkpoint row. Used on session complete/cleanup.

All methods use `Effect.fn('kargadan.checkpoint.{method}')` for tracing.

Per CONTEXT.md: "PostgreSQL checkpoint storage from Phase 1 — no intermediate local file mechanism. DB connection is a Phase 1 dependency."
