  <action>
    Create three new modules in `apps/kargadan/plugin/src/observation/`. Load `csharp-standards` skill. All modules use `namespace ParametricPortal.Kargadan.Plugin.src.observation;` and follow canonical section ordering. Mark all three with `[BoundaryAdapter]` since they interface with RhinoDoc events.

    **EventAggregator.cs** -- Channel-based event accumulation with debounced flush:

    Sealed class (not static -- holds mutable Channel and Timer state). Implements `IDisposable`.

    State:
    - `Channel<RawDocEvent> _channel` -- bounded to 256 entries, `DropOldest` full mode, `SingleWriter: false` (multiple event handlers), `SingleReader: true` (timer flush only).
    - `System.Timers.Timer _debounceTimer` -- 200ms interval (CONTEXT.md baseline), `AutoReset: false`.
    - `Action<EventBatchSummary> _onFlush` -- callback for delivering the flushed batch (injected via constructor).

    Public methods:
    - Constructor: `EventAggregator(Action<EventBatchSummary> onFlush)` -- creates channel, creates timer, wires timer.Elapsed to Flush.
    - `Enqueue(RawDocEvent rawEvent)` -> `bool`: Calls `_channel.Writer.TryWrite(rawEvent)`. Resets debounce timer (stop + start). Returns whether write succeeded.
    - `Flush()` -> `void`: Drains all available items from `_channel.Reader` via `TryRead` loop. If no items, returns without publishing. Groups into `EventBatchSummary` with `CategoryCount` and `SubtypeCount` breakdowns. Calls `_onFlush(summary)`.
    - `Dispose()`: Disposes timer, completes channel writer.

    Mark `[BOUNDARY ADAPTER -- timer callback and channel drain for event batching]` on the timer elapsed handler.

    Flush grouping logic:
    1. Drain channel into a `Seq<RawDocEvent>`.
    2. Group by `Type` (EventType) into `CategoryCount` entries.
    3. Within each category, group by `Subtype` (EventSubtype) into `SubtypeCount` entries.
    4. Set `ContainsUndoRedo` to true if any event has `IsUndoRedo == true`.
    5. Set `BatchWindowMs` to 200 (constant).

    **EventSubscriber.cs** -- subscribes to all RhinoDoc events:

    Sealed class. Constructor takes `EventAggregator aggregator`.

    Public methods:
    - `Subscribe()` -- subscribes to all 17 relevant RhinoDoc events (see research Code Examples section for full list):
      - Object events: `AddRhinoObject`, `DeleteRhinoObject`, `UndeleteRhinoObject`, `ReplaceRhinoObject`, `ModifyObjectAttributes`
      - Selection events: `SelectObjects`, `DeselectObjects`, `DeselectAllObjects`
      - Table events: `LayerTableEvent`, `MaterialTableEvent`, `DimensionStyleTableEvent`, `InstanceDefinitionTableEvent`, `LightTableEvent`, `GroupTableEvent`
      - Document events: `DocumentPropertiesChanged`
      - Command events: `Command.UndoRedo` (via UndoObserver), `Command.EndCommand` (NOT subscribed here -- ScriptRunner handles it per-call)
    - `Unsubscribe()` -- removes all event handlers (mirror of Subscribe).

    Each event handler:
    - Constructs a `RawDocEvent` with appropriate `EventType`, `EventSubtype`, optional `ObjectId`, optional `ObjectType`, and `IsUndoRedo` flag.
    - `IsUndoRedo` detected via `e.Document.UndoActive || e.Document.RedoActive` on object/layer events (confirmed in research).
    - Calls `_aggregator.Enqueue(rawEvent)` -- fast, non-blocking on UI thread (CRITICAL: research Pitfall 6).
    - No JSON serialization, no WebSocket writes, no blocking operations in handlers.

    For selection events (`SelectObjects`, `DeselectObjects`, `DeselectAllObjects`): use `EventType.SelectionChanged` with appropriate subtype. `DeselectAllObjects` has no ObjectId.
    For table events (`LayerTableEvent`, `MaterialTableEvent`, etc.): use the appropriate `EventType` variant. No ObjectId for table events.
    For `ReplaceRhinoObject`: capture both `e.ObjectId` (new) and `e.OldRhinoObject.Id` (old) in `RawDocEvent.OldObjectId`.
    For `ModifyObjectAttributes`: use `EventType.ObjectsChanged` with `EventSubtype.Modified`.

    **UndoObserver.cs** -- Command.UndoRedo event handler:

    Sealed class. Constructor takes `EventAggregator aggregator`.

    Public methods:
    - `Subscribe()` -- subscribes to `Command.UndoRedo += OnUndoRedo`.
    - `Unsubscribe()` -- removes handler.

    `OnUndoRedo` handler:
    - Per research Pattern 4: only emit on completion events (`IsEndUndo` or `IsEndRedo`), NOT on begin events.
    - Construct `RawDocEvent` with `EventType.UndoRedo`, subtype `EventSubtype.Undo` or `EventSubtype.Redo`.
    - Set `IsUndoRedo = true`.
    - Enqueue to aggregator.
    - Mark `[BOUNDARY ADAPTER -- Command.UndoRedo event handler for undo/redo detection]`.

    **EventPublisher.cs** -- extend to handle EventBatchSummary:

    The existing EventPublisher works with individual `EventEnvelope` records. Add a new method:
    - `PublishBatch(EventBatchSummary batch, Instant publishedAt)` -- serializes the batch summary as a single `EventEnvelope` with `EventType` set to a chosen category (or a new batch-level approach). ALTERNATIVE: Keep the existing `Publish(EventEnvelope, Instant)` and add a `PublishBatch(EventBatchSummary, Instant)` method that stores the batch summary in the queue as a wrapped `PublishedEvent`. The simplest approach: add a `PublishBatch` method that converts `EventBatchSummary` into a `JsonElement` delta and wraps it in an `EventEnvelope` with a new `EventId`, then calls existing `Publish`.

    Update KargadanPlugin to integrate the execution engine (Plan 01) and event observation pipeline (Task 1 above). Load `csharp-standards` skill.

    **BoundaryState extension:**
    Add `EventSubscriber`, `EventAggregator`, and `UndoObserver` to the existing `BoundaryState` record struct alongside the existing `EventPublisher`, `SessionHost`, and `WebSocketHost`.

    **OnLoad lifecycle:**
    In the existing `OnLoad` method, after creating `EventPublisher` and `WebSocketHost`:
    1. Create `EventAggregator` with a flush callback that:
       - Gets current `Instant` from `_timeProvider`.
       - Generates a new `EventId`.
       - Creates an `EventEnvelope` with the batch summary serialized as delta JSON.
       - Calls `EventPublisher.Publish(envelope, publishedAt)`.
       OR uses the new `EventPublisher.PublishBatch` method from Task 1.
    2. Create `EventSubscriber` with the aggregator.
    3. Create `UndoObserver` with the aggregator.
    4. Call `eventSubscriber.Subscribe()` and `undoObserver.Subscribe()`.
    5. Store all in `BoundaryState`.

    **OnShutdown lifecycle:**
    Before disposing WebSocketHost:
    1. Call `eventSubscriber.Unsubscribe()` and `undoObserver.Unsubscribe()`.
    2. Call `aggregator.Flush()` to emit any pending events.
    3. Dispose `aggregator`.

    **DispatchCommandAsync update -- two-phase response + execution routing:**

    Replace the current stub (`static (_) => Fin.Fail<CommandResultEnvelope>(Error.New(message: "Command execution not yet implemented."))`) with actual execution dispatch.

    The existing `DispatchCommandAsync` dispatches on UI thread via `ThreadMarshaler.RunOnUiThreadAsync`. Update the inner handler:

    1. Decode command envelope via existing `CommandRouter.Decode`.
    2. Route by `CommandOperation`:
       - `CommandOperation.RunScript`: Call `CommandExecutor.ExecuteRunScript(doc, envelope, commandScript, echo: true)` where `commandScript` is extracted from `envelope.Payload`. Payload must contain a `"script"` field with the command string. Extract via `envelope.Payload.GetProperty("script").GetString()`.
       - `CommandOperation.DirectApiCall`: Call `CommandExecutor.ExecuteDirectApi(doc, envelope, handler)` where `handler` dispatches to the appropriate `DocumentApi` method based on a sub-operation in the payload.
       - All existing read operations (`SceneSummary`, `ObjectMetadata`, etc.): Keep existing stub behavior for now (these are Phase 5+ concerns). Return `Fin.Fail` with "not yet implemented" message.
       - All existing write operations (`ObjectCreate`, `ObjectUpdate`, etc.): Route through `CommandExecutor.ExecuteDirectApi` with handlers that call `DocumentApi` methods.

    3. For write operations, extract parameters from `envelope.Payload` and dispatch to appropriate `DocumentApi` method:
       - `ObjectCreate`: Extract geometry type and parameters, construct `GeometryBase`, call `DocumentApi.CreateObject`.
       - `ObjectDelete`: Extract `objectId` from payload, call `DocumentApi.DeleteObject`.
       - `ObjectUpdate`: Extract `objectId` and new attributes, call `DocumentApi.ModifyAttributes`.
       NOTE: Full geometry construction from JSON payload is complex. For this phase, the dispatch structure must be wired but geometry construction from arbitrary JSON can use a placeholder that returns `Fin.Fail` with "geometry construction from payload not yet implemented" -- the actual geometry factories are an agent-level concern (Phase 5). The STRUCTURE must be complete: envelope decoded -> operation routed -> CommandExecutor wraps in undo -> DocumentApi called.

    4. Two-phase response pattern (CONTEXT.md locked decision):
       The existing `ThreadMarshaler.RunOnUiThreadAsync` already handles async dispatch. Add the ack phase:
       - Before marshaling to UI thread, send an "ack" response via the WebSocket (command started).
       - After UI thread execution completes, send the final result.
       - Implementation: The ack is a `CommandResultEnvelope`-shaped response with a new status indicating "started" -- OR, simpler: the WebSocketHost already sends responses. Add an `Ack` variant to indicate command receipt. The simplest approach for now: add a helper method `SendAckAsync` that writes a JSON `{ "_tag": "command.ack", "requestId": "..." }` message to the WebSocket before dispatching to UI thread. The WebSocketHost's send method is already available.

       NOTE: The two-phase ack may require access to the WebSocket send method from within the dispatch context. The existing `DispatchMessageAsync` receives the message but does not have direct WebSocket write access. The ack pattern should be wired at the WebSocketHost level (receive message -> send ack -> dispatch -> send result). If wiring the ack at the WebSocketHost level is too invasive, document the gap and defer the ack mechanism to integration testing.

    5. Per-command timeout (CONTEXT.md locked + Claude's discretion):
       The existing `DispatchCommandAsync` uses `WaitAsync(cancellationToken)`. Add per-command timeout from the envelope's `DeadlineMs` field:
       - Create a `CancellationTokenSource` with `CancelAfter(TimeSpan.FromMilliseconds(envelope.DeadlineMs))`.
       - Link with the existing `cancellationToken` via `CancellationTokenSource.CreateLinkedTokenSource`.
       - Wrap the `ThreadMarshaler.RunOnUiThreadAsync` call with `.WaitAsync(linkedCts.Token)`.

    The `HandleCommand` method signature already accepts `Func<CommandEnvelope, Fin<CommandResultEnvelope>> onCommand`. The wiring replaces the static stub with a real handler that reads `RhinoDoc.ActiveDoc` and routes to `CommandExecutor`.

    IMPORTANT: `RhinoDoc.ActiveDoc` provides the current document. All execution operates on the active document. Multi-document support is explicitly deferred (v2 ADVN-02).
