    Create three new modules in `apps/kargadan/plugin/src/observation/`. Load `csharp-standards` skill. All modules use `namespace ParametricPortal.Kargadan.Plugin.src.observation;` and follow canonical section ordering. Mark all three with `[BoundaryAdapter]` since they interface with RhinoDoc events.

    **EventAggregator.cs** -- Channel-based event accumulation with debounced flush:

    Sealed class (not static -- holds mutable Channel and Timer state). Implements `IDisposable`.

    State:
    - `Channel<RawDocEvent> _channel` -- bounded to 256 entries, `DropOldest` full mode, `SingleWriter: false` (multiple event handlers), `SingleReader: true` (timer flush only).
    - `System.Timers.Timer _debounceTimer` -- 200ms interval (CONTEXT.md baseline), `AutoReset: false`.
    - `Action<EventBatchSummary> _onFlush` -- callback for delivering the flushed batch (injected via constructor).

    Public methods:
    - Constructor: `EventAggregator(Action<EventBatchSummary> onFlush)` -- creates channel, creates timer, wires timer.Elapsed to Flush.
    - `Enqueue(RawDocEvent rawEvent)` -> `bool`: Calls `_channel.Writer.TryWrite(rawEvent)`. Resets debounce timer (stop + start). Returns whether write succeeded.
    - `Flush()` -> `void`: Drains all available items from `_channel.Reader` via `TryRead` loop. If no items, returns without publishing. Groups into `EventBatchSummary` with `CategoryCount` and `SubtypeCount` breakdowns. Calls `_onFlush(summary)`.
    - `Dispose()`: Disposes timer, completes channel writer.

    Mark `[BOUNDARY ADAPTER -- timer callback and channel drain for event batching]` on the timer elapsed handler.

    Flush grouping logic:
    1. Drain channel into a `Seq<RawDocEvent>`.
    2. Group by `Type` (EventType) into `CategoryCount` entries.
    3. Within each category, group by `Subtype` (EventSubtype) into `SubtypeCount` entries.
    4. Set `ContainsUndoRedo` to true if any event has `IsUndoRedo == true`.
    5. Set `BatchWindowMs` to 200 (constant).

    **EventSubscriber.cs** -- subscribes to 15 RhinoDoc events (UndoRedo handled by UndoObserver separately):

    Sealed class. Constructor takes `EventAggregator aggregator`.

    Public methods:
    - `Subscribe()` -- subscribes to exactly 15 RhinoDoc events:
      - Object events (5): `AddRhinoObject`, `DeleteRhinoObject`, `UndeleteRhinoObject`, `ReplaceRhinoObject`, `ModifyObjectAttributes`
      - Selection events (3): `SelectObjects`, `DeselectObjects`, `DeselectAllObjects`
      - Table events (6): `LayerTableEvent`, `MaterialTableEvent`, `DimensionStyleTableEvent`, `InstanceDefinitionTableEvent`, `LightTableEvent`, `GroupTableEvent`
      - Document events (1): `DocumentPropertiesChanged`
      NOTE: Command.EndCommand is NOT subscribed here -- ScriptRunner handles it per-call. Command.UndoRedo is NOT subscribed here -- UndoObserver handles it. Total across EventSubscriber (15) + UndoObserver (1 = Command.UndoRedo) = 16 subscriptions.
    - `Unsubscribe()` -- removes all 15 event handlers (mirror of Subscribe).

    Each event handler:
    - Constructs a `RawDocEvent` with appropriate `EventType`, `EventSubtype`, optional `ObjectId`, optional `ObjectType`, and `IsUndoRedo` flag.
    - `IsUndoRedo` detected via `e.Document.UndoActive || e.Document.RedoActive` on object/layer events (confirmed in research).
    - Calls `_aggregator.Enqueue(rawEvent)` -- fast, non-blocking on UI thread (CRITICAL: research Pitfall 6).
    - No JSON serialization, no WebSocket writes, no blocking operations in handlers.

    For selection events (`SelectObjects`, `DeselectObjects`, `DeselectAllObjects`): use `EventType.SelectionChanged` with appropriate subtype. `DeselectAllObjects` has no ObjectId.
    For table events (`LayerTableEvent`, `MaterialTableEvent`, etc.): use the appropriate `EventType` variant. No ObjectId for table events.
    For `ReplaceRhinoObject`: capture both `e.ObjectId` (new) and `e.OldRhinoObject.Id` (old) in `RawDocEvent.OldObjectId`.
    For `ModifyObjectAttributes`: use `EventType.ObjectsChanged` with `EventSubtype.Modified`.

    **UndoObserver.cs** -- Command.UndoRedo event handler:

    Sealed class. Constructor takes `EventAggregator aggregator`.

    Public methods:
    - `Subscribe()` -- subscribes to `Command.UndoRedo += OnUndoRedo`.
    - `Unsubscribe()` -- removes handler.

    `OnUndoRedo` handler:
    - Per research Pattern 4: only emit on completion events (`IsEndUndo` or `IsEndRedo`), NOT on begin events.
    - Construct `RawDocEvent` with `EventType.UndoRedo`, subtype `EventSubtype.Undo` or `EventSubtype.Redo`.
    - Set `IsUndoRedo = true`.
    - Enqueue to aggregator.
    - Mark `[BOUNDARY ADAPTER -- Command.UndoRedo event handler for undo/redo detection]`.

    **EventPublisher.cs** -- NO changes needed to EventPublisher itself:

    The existing `EventPublisher.Publish(EventEnvelope, Instant)` method is sufficient. Do NOT add a `PublishBatch` method. Instead, the flush callback provided to `EventAggregator` in `KargadanPlugin.OnLoad` is responsible for wrapping `EventBatchSummary` into an `EventEnvelope`:
    1. The callback captures `EventPublisher`, `SessionHost`, and `_timeProvider` from `KargadanPlugin.OnLoad` scope.
    2. On flush: get `Instant` from `_timeProvider`, generate `EventId` via `DomainBridge.ParseValueObject<EventId, Guid>(Guid.NewGuid())`, get session identity from `SessionHost` snapshot.
    3. Serialize `EventBatchSummary` as `JsonElement` delta via `JsonSerializer.SerializeToElement`.
    4. Call `EventEnvelope.Create(eventId, EventType.DocumentChanged, identity, sourceRevision, causationRequestId: None, delta, telemetryContext)`.
    5. Call `EventPublisher.Publish(envelope, publishedAt)`.
    This keeps EventPublisher generic and puts the batch-to-envelope mapping at the KargadanPlugin boundary where session identity is available.

    **DispatchMessageAsync signature change for two-phase ack (CONTEXT.md locked decision -- MUST be implemented, not deferred):**

    Pass a `Func<RequestId, Task> sendAckAsync` delegate from `WebSocketHost` into `DispatchMessageAsync`. The `WebSocketHost.ProcessMessageAsync` method already has access to the `WebSocket` instance. Create the delegate as a closure over the WebSocket that serializes `{ "_tag": "command.ack", "requestId": "..." }` and calls `webSocket.SendAsync`.

    Update the `MessageDispatcher` delegate signature:
    ```
    internal delegate Task<Fin<JsonElement>> MessageDispatcher(
        TransportMessageTag tag,
        JsonElement message,
        Func<RequestId, Task> sendAckAsync,
        CancellationToken cancellationToken);
    ```

    In `DispatchCommandAsync`, BEFORE calling `ThreadMarshaler.RunOnUiThreadAsync`, call `await sendAckAsync(requestId)` to send the ack immediately. Extract `requestId` from the message JSON. This satisfies the locked decision: ack is sent before UI thread dispatch, final result is sent after execution completes.

    **Execution routing:**

    Replace the current stub (`static (_) => Fin.Fail<CommandResultEnvelope>(Error.New(message: "Command execution not yet implemented."))`) with actual execution dispatch.

    The `onCommand` handler passed to `HandleCommand`:
    1. Read `RhinoDoc.ActiveDoc` for the current document. Multi-document support is deferred (v2 ADVN-02).
    2. Route by `CommandOperation`:
       - `CommandOperation.RunScript`: Extract `commandScript` from `envelope.Payload.GetProperty("script").GetString()`. Call `CommandExecutor.ExecuteRunScript(doc, envelope, commandScript, echo: true)`.
       - `CommandOperation.DirectApiCall`: Call `CommandExecutor.ExecuteDirectApi(doc, envelope, handler, onUndoRedo)` where `handler` dispatches to the appropriate `DocumentApi` method based on a sub-operation in the payload. The `onUndoRedo` callback is created in the dispatch closure, capturing `EventPublisher` and `SessionHost` to push undo notifications via `EventPublisher.Publish`.
       - `CommandOperation.ObjectCreate`: Route through `CommandExecutor.ExecuteDirectApi` with handler that calls `DocumentApi.AddPoint` for point payloads (the concrete factory from Plan 01). Other geometry types return `Fin.Fail` with "geometry construction from payload not yet implemented" -- full factories are Phase 5.
       - `CommandOperation.ObjectDelete`: Route through `CommandExecutor.ExecuteDirectApi` with handler that calls `DocumentApi.DeleteObject`.
       - `CommandOperation.ObjectUpdate`: Route through `CommandExecutor.ExecuteDirectApi` with handler that calls `DocumentApi.ModifyAttributes`.
       - All existing read operations (`SceneSummary`, `ObjectMetadata`, etc.): Keep `Fin.Fail` with "not yet implemented" message (Phase 5+ concerns).

    **Per-command timeout (CONTEXT.md locked + Claude's discretion):**
    - Extract `DeadlineMs` from the command envelope (default 30000ms if missing).
    - Create `CancellationTokenSource.CreateLinkedTokenSource(cancellationToken)`.
    - Call `linkedCts.CancelAfter(TimeSpan.FromMilliseconds(deadlineMs))`.
    - Wrap `ThreadMarshaler.RunOnUiThreadAsync` call with `.WaitAsync(linkedCts.Token)`.

    **File size concern:** KargadanPlugin.cs is currently 234 lines (just under 225-line cap pre-edit). Extract the command routing logic into a private static method `RouteCommand(RhinoDoc doc, CommandEnvelope envelope, BoundaryState state)` to keep the dispatch method small. The routing method is a `switch` expression on `envelope.Operation`.
