    **CommandManifestEntrySchema** -- Dual-purpose schema feeding both KB embedding (PERS-05) and future Tool.make definitions (Phase 5 AGNT-02/AGNT-04). Per user decision, metadata per command includes: parameters with types and defaults, natural language description, usage examples. Aliases and related commands are not prioritized.

    ```
    S.Struct({
        id:            S.NonEmptyTrimmedString,
        name:          S.NonEmptyTrimmedString,
        description:   S.NonEmptyString,
        params:        S.Array(S.Struct({
            name:        S.NonEmptyTrimmedString,
            type:        S.NonEmptyTrimmedString,
            description: S.optional(S.String),
            default:     S.optional(S.Unknown),
            required:    S.Boolean,
        })),
        examples:      S.Array(S.Struct({
            input:       S.NonEmptyString,
            description: S.optional(S.String),
        })),
        category:      S.optional(S.NonEmptyTrimmedString),
        isDestructive: S.optional(S.Boolean),
    })
    ```

    Derive type: `type CommandManifestEntry = typeof CommandManifestEntrySchema.Type`

    **CommandManifestSchema** -- Array wrapper: `S.Array(CommandManifestEntrySchema)`

    **loadManifest** -- Pure function that takes a `string` (JSON content), decodes via `S.decodeUnknown(S.parseJson(CommandManifestSchema))`, returns `Effect.Effect<ReadonlyArray<CommandManifestEntry>, ParseError>`. This is the loader used by the seeder.

    **SAMPLE_MANIFEST** -- A `const` array of 10-15 representative Rhino commands covering major categories. Per research, start with command names + descriptions (extractable from Rhino docs). Include params where known. Use commands that span categories:
    - Geometry creation: `Line`, `Circle`, `Box`, `Sphere`, `Extrude`
    - Modification: `Move`, `Copy`, `Rotate`, `Scale`, `Mirror`
    - Query: `What`, `Length`, `Area`, `Volume`
    - Viewport: `Zoom`, `Pan`

    Each entry should have: id (kebab-case slug), name (Rhino command name), description (1-2 sentences), params (known parameters with types), examples (1-2 usage examples), category, isDestructive flag.

    The manifest is intentionally static and extensible per user decision: "extraction pipeline transforms into a structured static manifest, then identify additional metadata fields the agent actually needs." Future phases will replace SAMPLE_MANIFEST with a full extracted manifest.

    Follow canonical file section order: SCHEMA -> CONSTANTS -> FUNCTIONS -> EXPORT. Use 4-space indentation.

<task type="auto">
  <name>Task 2: KB seeder service with embedding pipeline</name>
  <files>
    apps/kargadan/harness/src/knowledge/seeder.ts
  </files>
  <action>
    Create `apps/kargadan/harness/src/knowledge/seeder.ts` with:

    **KBSeeder** -- An `Effect.Service` class at `kargadan/KBSeeder` using `effect: Effect.gen(...)`. Yields `SqlClient.SqlClient` (for direct SQL against `search_documents` and `search_embeddings` tables -- per research, do NOT pull in DatabaseService which carries server-side deps the harness cannot provide).

    The service exposes a single `seed` method:

    `seed(manifest: ReadonlyArray<CommandManifestEntry>)` -- Takes the parsed manifest and:

    1. **Upsert search documents:** For each command, INSERT INTO `search_documents` with:
       - `entity_id`: command.id
       - `entity_type`: `'rhinoCommand'` (reuses existing search infra per research)
       - `scope_id`: `NULL` (global scope -- not tenant-scoped)
       - `content_text`: command.description
       - `display_text`: command.name
       - `metadata`: `{ params: command.params, examples: command.examples, category: command.category, isDestructive: command.isDestructive }` as JSONB
       - Use `ON CONFLICT (entity_type, entity_id) DO UPDATE SET ...` for idempotent re-seeding
       - Concurrency: `Effect.forEach(manifest, ..., { concurrency: 10, discard: true })`

    2. **Generate embeddings:** Build embedding text for each command as `[command.name, command.description, ...command.params.map(p => p.name)].join(' ')`. Collect all texts into an array. The embedding generation must go through an external API per user decision ("Embedding model: external API"). Since the harness cannot depend on the full AiRuntime service (server-side deps), implement embedding as a direct Effect wrapping the OpenAI embeddings API via `@effect/ai-openai` `OpenAiClient`:
       - Yield `OpenAiClient.OpenAiClient` from the service constructor
       - Call `client.client.embeddings.create({ input: texts, model: 'text-embedding-3-small', dimensions: 1536 })` in batches of 100
       - Actually -- SIMPLER: since the harness already has `@effect/ai` and the seeder is a build-time/one-time script, accept an `embed` function parameter: `embed: (texts: ReadonlyArray<string>) => Effect.Effect<ReadonlyArray<ReadonlyArray<number>>, unknown>`. This keeps the seeder decoupled from any specific AI provider. The caller provides the embed function (Phase 5 wires the real AiRuntime; for now, the function signature is established).

    3. **Upsert embeddings:** For each command+embedding pair, INSERT INTO `search_embeddings` with:
       - `entity_type`: `'rhinoCommand'`
       - `entity_id`: command.id
       - `scope_id`: `NULL`
       - `model`: fixed string from config (default `'text-embedding-3-small'`)
       - `dimensions`: 1536
       - `embedding`: the vector as `halfvec` (using `sql`...`::halfvec(3072)` with zero-padding to 3072 matching existing table schema)
       - `document_hash`: `hashCanonicalState(command)` from checkpoint.ts (reuse existing hash function)
       - Use `ON CONFLICT (entity_type, entity_id, scope_id, model, dimensions) DO UPDATE SET ...` for idempotent re-embedding
       - Concurrency: `Effect.forEach(pairs, ..., { concurrency: 10, discard: true })`

    4. **Return summary:** `{ documentsUpserted: number, embeddingsUpserted: number }`

    The `embed` parameter makes the seeder testable without real API calls. The `hashCanonicalState` import from `../persistence/checkpoint.ts` reuses the existing canonical hashing function (already exported).

    Per user decision: "Ranking: pure pgvector cosine similarity -- closest embedding wins, no usage frequency boosting." The seeder does not implement ranking -- that is handled by the existing `search_embeddings` HNSW index with `halfvec_cosine_ops`.

    Follow canonical file section order: TYPES -> SCHEMA -> SERVICES -> EXPORT. Use 4-space indentation.
